Shogun 2D

Internal documentation — engine architecture, controls reference, and system breakdowns.

Architecture

Shogun 2D is built on Phaser 3.87 with Arcade Physics. The project uses Vite for bundling and TypeScript throughout.

Phaser.Game ├─ BootScene Load all assets, create animations, then start Game └─ GameScene Main gameplay loop ├─ LevelBuilder Procedural level geometry + decorations ├─ PlayerController Movement, input, physics, animation └─ DevPanel Runtime parameter tuning (DOM overlay)
GAME CONFIG
Resolution: 960 x 540 · Pixel art mode enabled · Arcade physics with gravity 800 · Scale mode: FIT + center

Project Structure

src/ main.ts — Phaser.Game bootstrap + config scenes/ BootScene.ts — Asset loading + animation creation GameScene.ts — Main game loop, colliders, camera player/ PlayerController.ts — All player logic, input, and config level/ LevelBuilder.ts — Ground, platforms, walls, decorations debug/ DevPanel.ts — Runtime slider UI for tuning public/ assets/ — Sprites, backgrounds, tilesets docs.html — This documentation page index.html — Game entry point

Scene Flow

BootScene.preload() │ Load 22 samurai sprite sheets (96×96 frames) │ Load 3 parallax background layers │ Load tileset, trees, props, objects, flag sheet │ Show progress bar during load ▼ BootScene.create() │ Create 10 player animations + flag animation │ Transition to Game scene ▼ GameScene.create() │ Build level geometry (LevelBuilder) │ Spawn player at configured spawn position │ Register colliders: player ↔ platforms, player ↔ walls │ Configure camera bounds + follow player │ Create DevPanel (if devMode) ▼ GameScene.update(delta) ← runs every frame │ PlayerController.update(delta) │ Check fall-off-world respawn │ DevPanel.update() (refresh stats display)

Movement System

The player controller runs a state-based movement system each frame. The priority order is:

1. Apply gravity scale 2. Ground check → body.blocked.down || body.touching.down 3. Wall check → body.blocked.left || body.blocked.right 4. Update timers → coyote, jump buffer, wall-jump lock, dash cooldown 5. Dash logic → if dashing, maintain velocity and skip movement 6. Horizontal → acceleration-based movement toward target speed 7. Jumping → coyote time + buffer; wall jump; variable height 8. Fall gravity → increased gravity when falling (fall multiplier) 9. Wall slide → cap downward velocity when sliding 10. Dash trigger → initiate dash, disable gravity 11. Animation → select animation from movement state

Acceleration Model

Horizontal movement uses separate acceleration and deceleration values for ground and air:

targetSpeed = inputX * moveSpeed * (running ? runMultiplier : 1)
accel = grounded ? (moving ? acceleration : deceleration)
                 : (moving ? airAcceleration : airDeceleration)
diff = targetSpeed - currentVelocityX
change = accel * (delta / 1000)
newVelocity = currentVelocityX + sign(diff) * min(change, abs(diff))
NOTE
The wall jump lock timer prevents horizontal input from overriding wall jump velocity for a short window, giving the wall jump a committed arc.

Jump Mechanics

Coyote Time

When the player walks off a ledge, a timer starts counting down from coyoteTime (default 80ms). During this window, pressing jump still works as if grounded — essential for platforming feel.

Jump Buffering

When jump is pressed mid-air, a buffer timer starts at jumpBufferTime (default 100ms). If the player lands before the buffer expires, the jump executes automatically. Both systems combine: you can walk off a ledge and buffer a jump during coyote time.

Variable Jump Height

Releasing the jump key while ascending multiplies upward velocity by jumpCutMultiplier (default 0.4). This gives a short hop on tap and full jump on hold.

Fall Multiplier

When velocity.y > 0 (falling), gravity is multiplied by fallMultiplier (default 1.5). This makes the fall arc feel snappier and more responsive than a symmetric parabola.

Wall Mechanics

Wall Slide

Conditions: not grounded + touching wall + falling + holding directional input toward the wall. When active, downward velocity is capped to wallSlideSpeed.

Wall Jump

Press jump while touching a wall (and not grounded) to wall jump. The player receives:

  • Upward velocity: -wallJumpForce
  • Horizontal velocity: -wallDirection * wallJumpHorizontalForce (away from wall)
  • Facing flips to face away from the wall
  • Input is locked for wallJumpLockTime to commit the arc

Dash Mechanics

Press X (or B button on mobile) to dash. Gravity is disabled during the dash. The player moves at dashSpeed in the facing direction for dashDuration ms. A cooldown of dashCooldown ms prevents spamming. During the dash, all other movement input is skipped — the update loop returns early after the animation step.

Collision System

Collision uses Phaser's Arcade Physics with StaticGroup bodies.

Two Collision Groups

GroupContainsPurpose
platformsGround segments + floating platformsWalking, landing, ground detection
wallsVertical wall segmentsWall slide, wall jump detection

How Collision Bodies Work

Each level element has two parts:

  1. Visual: A TileSprite for rendering (scaled 2x from 16px art)
  2. Physics: An invisible StaticBody in the appropriate group with explicit size via setSize + setOffset

Colliders are registered in GameScene.create():

this.physics.add.collider(player.sprite, level.platforms);
this.physics.add.collider(player.sprite, level.walls);

Player Collision Box

The player sprite is 96x96 but the physics body is trimmed to 24x40 with offset (36, 50), fitting tightly around the character's torso and legs.

Parameter Sliders

Grouped by category (Horizontal, Jump, Wall, Dash, Physics, Spawn). Each slider updates the corresponding MovementConfig value in real time via setConfigValue(). The "Reset All to Defaults" button restores original values.

NOTE
Slider changes are not persisted. Refreshing the page resets all values to defaults defined in DEFAULT_CONFIG.

Serialized Fields

The SerializedField interface describes each tunable parameter:

interface SerializedField {
  key: string;       // Config property name
  label: string;     // Human-readable label
  value: number;     // Current value
  min: number;       // Slider minimum
  max: number;       // Slider maximum
  step: number;      // Slider increment
  category: string;  // Grouping in the dev panel
}

getSerializedFields() returns the full array (22 fields). setConfigValue(key, value) writes back. Special handling: changing maxFallSpeed also calls body.setMaxVelocityY() for immediate effect.
